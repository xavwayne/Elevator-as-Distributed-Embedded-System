Verification Framework Description

In the RuntimeMonitor base class, there is a large amount of physical payloads that covers all the physical payloads we may need to monitor the whole system and check whether the design matches the high level requirements, if not ,warning will be thrown. On the other hand, What we do for monitoring is to use state machines such as CarCallStateMachines, or  functions like updateCurrentFloor() to derive events about high level requirements and when related requirements violations events happens, those state machine will call other monitoring functions,e.g, doorOpening(),  to detect and record warnings, and updateCurrentFloor() function will implement the monitoring itself.

RT-6 The Car shall only stop at Floors for which there are pending calls.


Transition #
 Guard Condition
[R-T6.T1] DriveSpeed != Stop
[R-T6.T2] DriveSpeed == Stop && (CarCall[f,b] == True || HallCall[f,b,d] == True)
[R-T6.T3] DriveSpeed != Stop
[R-T6.T4] DriveSpeed == Stop && (CarCall[f,b] == False && HallCall[f,b,d] == False)

When the elevator stops, the DriveSpeedStateMachine of monitor enters the Car_STOP warning state if CarCall[f,b] and HallCall[f,b,d]  both equal to false, where b equals to any possible value of current floor and f equals to current f, It means that there is no pending calls. otherwise it enters the normal stop state. When moving the monitor returns to the state Car_MOVE.

RT-7 The Car shall only open Doors at Hallways for which there are pending calls.

Transition # Guard Condition
[R-T7.T1] DoorClosing == True
[R-T7.T2] DoorOpening == True && (HallCall[f,b,d] == True || CarCall[f,b] == True)
[R-T7.T3] DoorClosing == True
[R-T7.T4] DoorOpening == True && (HallCall[f,b,d] == False && CarCall[f,b] == False)

When the Door starts opening, the DoorStateMachine of monitor enters the doorOpening state. if CarCall[f,b] and HallCall[f,b,d]  both equal to false, It means that there is no pending calls, then the external function of DoorStateMachine will throw a warning. When door starts closing the monitor returns to the state Door_Close.

RT-8.1 If any door is open at a hallway and there are any pending calls at any other floor(s), a Car Lantern shall turn on.

Transition # Guard Condition
[R-T8.1.T1] DoorOpening == True && (any CarCall[f,b] == False && HallCall[f,b,d] == False where f != current floor)
[R-T8.1.T2] DoorClosing == True
[R-T8.1.T3] DoorOpening == True && (HallCall[f,b,d] == True || CarCall[f,b] == True)
[R-T8.1.T4] DoorClosing == True
[R-T8.1.T5] DoorClosing == True
[R-T8.1.T6] DoorOpening == True && (HallCall[f,b,d] == True || CarCall[f,b] == True)
When the door starts opening, the DoorStateMachine of monitor enters the doorOpening state. if any CarCall[f,b] or HallCall[f,b,d]  equal to false, where f is not current floor,   It means that there is  pending calls at other floors,and if no Car Lantern is On, then the external function of DoorStateMachine will throw a warning.
Meanwhile, CarLanternStateMachine keeps updating the state of car lantern.

RT-8.2 If one of the car lanterns is lit, the direction indicated shall not change while the doors are open.  


Transition # Guard Condition
[R-T8.2.T1] CarLantern[d] == True
[R-T8.2.T2] CarLantern[d] == False
[R-T8.2.T3] CarLantern[d] == False && all DoorClosed != True
[R-T8.2.T4] CarLantern[d] == True && all DoorOpening != True
[R-T8.2.T5] CarLantern[d] == True
[R-T8.2.T6] CarLantern[d] == False
When the car lantern changes state, the CarLanternStateMachine enters On or Off state, If CarLantern Change state from on to off or from off to on,  external functions lanternOff() or lanternOn will be called to check whether this change happens if the doorstate is not fully closed or not in opening state respectively. Both of those cases will throw a warning because the direction indicated change while the doors are open.

RT-8.3  If one of the car lanterns is lit, the car shall service any calls in that direction first.



Transition # Guard Condition
[R-T8.3.T1] Drive.Direction != prevCarLanternState && any CarCall or HallCall ==true in that direction
[R-T8.3.T2] Drive.Direction == STOP || Drive.Direction == prevCarLanternState
Whenever the CarLantern change state from off to on, the external function lanternOn will be called and reltated car lantern direction will be recorded as the prevCarLanternState. Once the car starts to moving, the DriveStateMachine will enters state of related moving direction such as UP, DOWN, and if this direction doesn't match the prevCarLanternState, a warning will be throw.

RT-9 The Drive shall be commanded to fast speed to the maximum degree practicable.

Transition # Guard Condition
[R-T9.T1] Drive.speed() > 0
[R-T9.T2] Drivespeed.speed() == 0 && fastSpeedCommand == true
[R-T9.T3] Drive.speed() > 0
[R-T9.T4] Drivespeed.speed() == 0 && fastSpeedCommand == false

When the car stops at some floor, the carStop() function will check whether the fastSpeedCommand has been set to true or false, if true, it means that once during the traveling state the car has been commanded to  fast speed to the maximum degree.Otherwise it was not commanded to fast speed. Once the all doors are closed, the doorClosed() function will reset boolean fastSpeedCommand to false again.

RT-10  For each stop at a floor, at least one door reversal shall have occurred before the doors are commanded to nudge.



Transition # Guard Condition
[R-T10.1] DoorCommand == NUDGE && reversalCount >= 1
[R-T10.2] DoorCommand == NUDGE && reversalCount == 0
[R-T10.3] DoorClosed == true
[R-T10.4] DoorClosed == true
[R-T10.5] DoorCommand = Open
[R-T10.6] CountDown <= 0
[R-T10.7] isReversing == true
[R-T10.8] DoorClosed == true
When the door reversal is treggered, reversal count will plus 1, and reversal count will be set to 0 when the door is fully closed. Once the monitor received nudge message, monitor will check the count, i.e., whether the door was triggered before, if yes, reversal count is larger than 0 otherwise equals to 0. So if count is equal to zero and doormotor is in nudge command, we will record the invalid nudge state and throw a warning about that .
